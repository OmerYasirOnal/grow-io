# grow.io v1.3 — "Worlds & Progression" — Design

> Status: **proposed (awaiting approval)** · 2026-06-02
> Grounded in a 5-lens code assessment of the shipped game (architecture, visuals/juice,
> progression/meta, modes, content/monetization). File:line anchors below refer to
> `game/index.html` unless noted.

## 1. Vision

Turn grow.io from a **single endless arena with cosmetic-only meta** into a **multi-mode game with
a real progression spine, a fixed coin economy, and a premium neon feel** — while staying a
**single-file Canvas2D game** loaded by the Expo WebView shell.

One organizing idea ties all four pillars together: **every run now feeds permanent forward motion
AND offers a chosen goal.** A 200-score death still earns XP toward a level; a daily mission shapes
how you play; coins finally buy things worth grinding for; and the moment-to-moment game *feels*
premium.

**Why now (owner intent):** the current build feels too shallow / "meaningless" and needs to be
both deeper and more beautiful. v1.3 deliberately spans all four requested pillars: in-game
visuals & juice, depth & progression, new modes, content & monetization.

## 2. Architecture constraints (hard — these shape every feature)

- The **whole game is one ~2293-line `'use strict'` classic script** (`game/index.html`, strict at
  :30). No `import`/`export`. Top-level `function` decls land on `window` (the native bridge relies
  on this); `let`/`const` globals do **not**. **A single JS error blanks the entire game.**
- It is **loaded as a file:// asset** by the Expo shell (`src/GameWebView.tsx:16-28`). **Do NOT
  modularize to ES modules** — it would break the load path and the bridge. Keep it single-file;
  the real risk is disorganized growth, not the single file.
- **Persistence is dual-layer, local-authoritative** (localStorage wins; native AsyncStorage is a
  first-load floor). Coins use a proven pure reconcile (`reconcileLoadedCoins` :388). New persisted
  data must round-trip through both and **never corrupt an existing user's save on update**.
- **Animation convention:** UI animates off `frameCount`; `gdt` (frame delta in **seconds**) is for
  frame-rate-independent motion. **No per-frame allocation** — the game runs up to 12 bots × 60fps ×
  up to 4 substeps; reuse module buffers (`liveBotsBuf` :1219 pattern, `.length=0` + refill).
- **Native bridge** = `postMsg` out + one `message` listener in, 8 message types (HAPTIC; SAVE/LOAD
  for HIGH_SCORE/COINS/STATS/SKIN_STATE; ad messages). **Extend by piggybacking existing types
  (esp. STATS), not by adding native message types**, so the JS can ship without a native rebuild.
- The codebase is **strictly sectioned by `// ═══ NAME ═══` banners** in a fixed order
  (constants → globals → audio → bridge → persistence → grid → particles → food → powerups → snakes
  → update → render → draw helpers → input → loop). **Every new system follows this contract:** its
  own banner block, pre-allocated buffers + state near the globals, an `update*(dt)` wired into
  `update()`, a `draw*()` wired into the render switchboard, save/load in the persistence block.

## 3. The four pillars → features

### Pillar A — In-game visuals & juice (make moment-to-moment play feel premium)
The menu just got a glow-up; the **arena** has not. Today a **kill and eating a pellet use the same
flat-dot primitive** (20 vs 3 dots), boost is nearly invisible on the snake, and 5 "zones" are just
a background color swap.
- **Radial shockwave + additive-glow burst** on kill / death / boss-slay (new pool beside `spawnP`
  :505-538; fired from `killSnake` :882/:900). Additive (`'lighter'`) compositing, bounded to the
  effects layer, is the trick that makes neon read as premium.
- **Boost feel** — head speed-lines + body/head glow pulse + a brighter multi-segment trail, gated
  to the player while boosting (`drawSnake` head :1378, boost trail :1215).
- **Zones → biomes** — cached radial-gradient bg + `zoneAccent`-tinted boundary/grid + an expanding
  accent ring on zone entry; enrich the `ZONES` table (:141) with per-zone fields (food density,
  hazard/bot bias). Depth-progression becomes *felt* (`updateZone` :1057 already ticks off score).
- **Micro-polish (P2):** living eyes (pupils track + blink), eat head-punch/absorb-streak, boundary
  danger glow near the rim.

### Pillar B — Depth & progression / meta (the spine; the direct fix for "meaningless")
The game has stats but **no loop that converts a run into permanent motion** — progression
dead-ends at "bought all skins."
- **Account XP + Level + a MILESTONES reward track** (coins / exclusive cosmetic / charges at
  levels 3,5,8,12…). XP awarded in the death block (`killSnake` :902) so even a bad run is
  productive. Thin XP bar on menu + level-up toast (reuse toast machinery).
- **Daily challenges** — 3 date-seeded rotating missions (reach score / N kills in a run / slay a
  King / survive T) paying coins + XP, with an all-3 bonus. Read from existing counters; reset via
  `todayKey` (:401).
- **Named score tiers** (Bronze→Silver→Gold→Platinum→Diamond→Master) derived from `highScore`
  (:328) with a "next tier at N" nudge. Near-free, pure additive.
- **Achievements (P2)** — permanent badges reading the already-saved lifetime stats
  (`totalKills`/`totalGames`/`deathCount`, currently dead data at :1756).

### Pillar C — New modes (variety; turn one arena into a mode platform)
- **Foundation:** a `mode` global + `MODES` config table **orthogonal to the `state` screen
  machine**, a `'modeselect'` screen mirroring `'leaderboard'`/`'skinselect'`, and a parameterized
  `startGame(modeId)`. Branch only the 3-4 seams that already vary per run (`startGame` :1066,
  `update` win/lose :1136, `killSnake` end-scoring :898, `drawHUD` :1442). After this, a new mode is
  ~20-40 lines. Default `mode='classic'` → existing behavior byte-for-byte unchanged.
- **Rush** — 90s score sprint (escalated spawn cadence + pulled-in King/Boss), HUD timer, ends via
  the existing death path. Cheapest first mode.
- **Per-mode best scores** (`bestByMode` map) + a mode badge on the death screen / leaderboard
  header — extends, never replaces, the existing `growio_hs`/`growio_top` keys.
- **Boss-Rush (P2)** — back-to-back bosses, scored by waves killed; reuses the boss block
  (:1192-1201). Value compounds after Rush proves the framework.

### Pillar D — Content & monetization (give coins a reason to exist; add revenue surfaces)
The assessment found a **real economy leak**: `checkUnlocks()` (:1040) hands out **for free** the
same skins the shop charges coins for (:2231) — so coins, the shop, the daily bonus, and the
rewarded-coin ad all have near-zero value, which suppresses the highest-eCPM unit.
- **Fix the dual-unlock leak** — feat skins (score/kill/length/games tiers) become **earn-only** (no
  Buy button); the rest become **coins-only**; retune `skinPrice` (:417) to a real 2-4 week curve.
  Migration-safe: owned skins stay (`loadSkins` filter :362 preserves owned indices). *This is the
  lever that makes every other monetization surface matter — it lands before them.*
- **"Double your coins" rewarded ad** on the death screen (the screen already computes `earned`
  :1903 but offers no action). Highest-converting placement in the genre; reuses the existing
  `SHOW_REWARDED_AD` path + `AD_REWARD_EARNED` switch (:318) with a new `reward:'double'` string.
- **Scaling menu rewarded** — replace the flat "+100 coins" (:320, meaningless vs a 6000-coin skin)
  with a wheel/scaling payout + a "next skin in N coins" progress hint.
- **Tappable escalating 7-day daily-reward modal** + ad-funded streak-save — replaces the passive
  un-interactable toast (`checkDailyBonus` :404). Strong D1/D7 lever; streak data already persisted.
- **Coins-only TRAILS cosmetic axis** (sparkle/embers/neon-streak/bubbles/rainbow-dust) selectable
  independently of skin — pure particle code (CC0-safe), doubles catalog depth, adds coin sinks.
- **Featured rotating shop slot (P2)** — 1-2 weekly deterministic (week-of-year) items + "NEW"
  badge; turns a static grid into a habit loop. No backend.

## 4. Roadmap (waves & priorities)

**WAVE 0 — Foundation (P0, ships first, own PRs):**
1. `saveBlob/loadBlob(key, defaults)` versioned additive save helper (routed through the existing
   **STATS** bridge — **zero new native message types**), validated in isolation like
   `reconcileLoadedCoins`. Everything persisted in v1.3 goes through it.
2. `mode` global + `MODES` config + `'modeselect'` screen + `startGame(modeId)`; codify the
   section/buffer/strict-global conventions in a header comment.

**WAVE 1 — Spine + economy + core juice (P0, the shippable heart):**
- XP/Level + MILESTONES · Economy-leak decouple · Shockwave + additive particles · Rush mode +
  per-mode bests.

**WAVE 2 — Retention + felt-depth (P1):**
- Daily challenges · "Double coins" rewarded + scaling menu reward · 7-day daily-reward modal +
  streak-save · Boost juice · Zones→biomes · Score tiers · Trails axis.

**WAVE 3 — Long-tail (P2):**
- Boss-Rush · Living-eyes/eat/boundary micro-juice · Achievements · Featured shop slot.

**Explicitly OUT of v1.3 (next release):** IAP / Remove-Ads (the one item needing a genuinely new
native surface — sequence *after* the fixed economy is proven live), Daily-Seed & Horde modes, an
upgrade/boost tree, a slow-mo death-cam.

## 5. Critical-path dependencies

```
saveBlob ───────────► everything persisted (xp/level, missions, per-mode bests, trails, daily-claim)
MODES config ───────► Rush ──► Boss-Rush ; per-mode bests
economy-leak fix ───► double-coins, daily ladder, trails, featured shop
additive particles ─► boost juice, trails, biome entry sweep
```

## 6. Risks & mitigations

1. **A single JS error blanks the whole game.** → Contract: every global `let`/`const`; each system
   in its own banner block wired at the known seams; a dev-assert / CI grep flagging bare top-level
   assignments and per-frame allocation; defensive `draw*()` bodies where cheap; one P0 system per
   PR, smoke `menu→play→death→menu` before stacking the next.
2. **Corrupting live saves on update.** → Route ALL new data through `saveBlob` (Wave 0); reuse the
   local-authoritative + first-load-floor reconcile for spendables; piggyback the STATS bridge (no
   new native types); **test the upgrade path on a device that already has v1.2 data.**
3. **Per-frame allocation / fillrate regression** (12 bots × 60fps × 4 substeps). → Reused-buffer
   pattern for every new pool; cache the gradient bg (regenerate only on zone settle); bound the
   additive pass to effects; cap new pools; profile on a real low-end device, not the simulator.
4. **Scope creep.** → P0 (foundation + XP + economy-fix + shockwave/additive + Rush + per-mode
   bests) is the shippable core; P1/P2 are cut-able. Ship P0 the moment it's stable.
5. **Economy-retune backlash.** → Grandfather all owned/earned skins; tune the new curve so banked
   coins still buy something day one; frame as "new things to spend on," not a takeaway.
6. **Ad/UX fatigue.** → All new rewarded surfaces opt-in & contextual; keep the 60s interstitial
   cooldown; never gate progression behind an ad (XP/missions pay free; ads only multiply).

## 7. Build & ship process

- **Integration branch:** `feat/v1.3-worlds-progression` (off `main`). Each feature = a sub-branch →
  PR into the integration branch → **fresh zero-context adversarial review → green → merge**. When
  v1.3 is stable, the integration branch → `main`.
- **Verification per PR:** `npm run typecheck` clean · `npm test` · browser smoke of the affected
  screen (a JS error halts everything, so "functions defined + screen renders" is a strong signal).
- **After Wave 0:** verify on a real binary that an existing user's coins/skins/high-score survive
  the update untouched, before stacking features.
- **Public release:** Ömer does the final **Xcode archive → Distribute** himself (not EAS
  submit-to-review). This effort gets the code + a testable build to that point.
